## TL;DR

今回やったことは以下です。

- Raspberry Pi 5 に **Tailscale** を入れて、ポート開放ゼロで外出先からの SSH を 5 分で実現しました
- 公式 `install.sh` が **Raspberry Pi OS 13 (Debian trixie)** を自動判定してくれました
- `--ssh` フラグを付けるだけで **SSH 鍵の配布も不要** になります(Tailscale SSH)
- MagicDNS で `ssh hostname` のように **ホスト名だけで届く** ようになります
- 常時稼働サーバーは **Key Expiry を Disable** にしておくのが定石です

![Tailscale Admin の machines 一覧 — 4 台がメッシュに参加している](/images/1479b0552927e1/01-tailnet-machines.png)

## はじめに

[前回の記事](https://zenn.dev/harieshokunin/articles/9caa728de35b1f) で Raspberry Pi 5 (`hostname.local`) の初期セットアップを終えました。今回はその続編として、**外出先からも自宅の RPi5 に安全にアクセスできる状態** を作っていきます。

定期実行・ブラウザ自動化・通知などの軽量タスクを RPi5 に任せたかったので、LAN 内に居ないと触れないままだと不便すぎました。かといって、自宅ルーターのポートを世界に向けて開けるのは怖いです。

そこで今回は、**ポート開放なしで NAT を貫通できる Mesh VPN** を入れる方針にしました。

## なぜ Tailscale を選んだか

主要候補をいくつか比較した結果、Tailscale が一番手軽でした。

| サービス | 料金 (2026年5月時点) | NAT越え | セットアップ | 備考 |
|---------|-------------------|--------|-----------|------|
| **Tailscale** | 個人無料(〜100台 / 3ユーザー) | ◎ DERP 中継 | 1コマンド | WireGuard ベース。`tailscale ssh` で鍵管理不要 |
| ZeroTier | 無料(〜25台/ネットワーク) | ◎ | 中 | レイヤ2 VPN |
| Cloudflare Tunnel | 完全無料 | ◎ | 中 | tunnel 方式、L4限定 |
| PiVPN (WireGuard 自前) | 無料 | △ ポート開放必須 | 高 | CGNAT環境では詰む |
| OpenVPN | 無料 | △ | 高 | レガシー |

Tailscale を選んだ決め手はこの 3 つです。

1. **ルーターのポート開放が不要** — CGNAT 配下でも DERP 中継で貫通します
2. **`tailscale ssh` で SSH 鍵配布から解放される** — Tailnet の認証がそのまま SSH 認証になります
3. **MagicDNS で `ssh hostname` だけで届く** — `.local` も IP も覚えなくていいです

> Tailscale のコントロールプレーン(座標サーバー)は商用クラウドに依存します。完全自前でやりたい人は OSS 互換実装の [Headscale](https://github.com/juanfont/headscale) を選択肢に入れてください。今回は手早く動かすことを優先して Tailscale を採用しました。

## セットアップ

### 1. Tailscale をインストールする

公式インストーラを使います。**Raspberry Pi OS 13 (Debian trixie) を自動判定** してくれます。

```bash
curl -fsSL https://tailscale.com/install.sh | sh
tailscale version
```

実行ログはこんな感じでした (抜粋)。

```
Installing Tailscale for debian trixie, using method apt
+ curl -fsSL https://pkgs.tailscale.com/stable/debian/trixie.noarmor.gpg
...
Setting up tailscale (1.96.4) ...
Created symlink '/etc/systemd/system/multi-user.target.wants/tailscaled.service' → '/usr/lib/systemd/system/tailscaled.service'.
Installation complete!
```

### 2. 起動 + Tailscale SSH を有効化する

```bash
sudo tailscale up --ssh --hostname=hostname
```

| オプション | 意味 |
|-----------|------|
| `--ssh` | Tailscale SSH を有効化(SSH鍵不要、Tailnet認証で接続) |
| `--hostname=hostname` | MagicDNS 用のホスト名を明示 |

実行すると `https://login.tailscale.com/a/xxxxxx` が表示されます。手元の PC ブラウザで開いてアカウントで承認すると、hostname が Tailnet に参加します。

### 3. 接続確認

```bash
tailscale ip -4
# → 100.72.2xxxxx

tailscale status
# 100.72.2xxxxx    hostname             me@  linux    -
# 100.67.2xxxxx    iphone174            me@  iOS      -
# 100.80.1xxxxx    XXXXXXXXX-2          me@  macOS    -
```

Tailscale 管理画面 (login.tailscale.com/admin/machines) にも `hostname` が **`SSH` バッジ付き** で現れます。さらに後述の Key Expiry を無効化すると `Expiry disabled` バッジも追加されます。

### 4. 手元の Mac からアクセスする

`.local` も IP も書きません。**MagicDNS でホスト名だけで届きます**。

```bash
# 通常の SSH (MagicDNS 経由)
ssh user@hostname

# Tailscale SSH (鍵もパスワードも不要)
tailscale ssh user@hostname
```

![Mac メニューバーの Tailscale から Tailnet 内デバイスを選んで接続](/images/1479b0552927e1/02-mac-tailscale-menu.png)

初回は次のような追加認証が走ります。

```
# Tailscale SSH requires an additional check.
# To authenticate, visit: https://login.tailscale.com/a/xxxxx
# Authentication checked with Tailscale SSH.
```

これは Tailscale のデフォルト ACL の **`check` モード** の挙動で、Tailnet のオーナー自身でも 12時間ごとにブラウザ確認が要求されます (詳細は後述します)。

## ハマりどころ

### Raspberry Pi OS 13 は Bookworm ではなく Trixie

`/etc/os-release` を見るとわかりますが、**Raspberry Pi OS 13 のベースは Debian trixie (13)** でした。Bookworm 前提で書かれた古い記事のコマンドをそのままコピペすると、`pkgs.tailscale.com/stable/raspbian/bookworm` を見に行ってしまって詰むケースがあります。

公式 `install.sh` は trixie を自動判定して `pkgs.tailscale.com/stable/debian/trixie` を使ってくれるので、迷ったら `install.sh` 一択でした。

### locale 警告 `LC_CTYPE = "ja_JP.UTF-8"` が消えない

Mac の SSH クライアントは `SendEnv LC_*` で Mac 側のロケールを hostname に送ってきます。hostname 側で `ja_JP.UTF-8` が **生成されていない** と、apt 実行のたびに `perl: warning: Setting locale failed.` が連発しました。

`locales-all` パッケージを入れただけでは足りず、**`/etc/locale.gen` の有効化と `locale-gen` の実行が別途必要**でした。

```bash
sudo sed -i 's/^# *\(ja_JP.UTF-8 UTF-8\)/\1/' /etc/locale.gen
sudo locale-gen
locale -a | grep -i ja_JP   # ja_JP.utf8 が出ればOK
```

これで新しい SSH セッションから警告は消えました。

### `ssh hostname` で `Tailscale SSH requires an additional check` が出る

これは特に設定した覚えがなくても出ます。**新規 Tailnet を作ると Tailscale 側がデフォルト ACL を自動で投入** していて、その中の `ssh` セクションが `check` モードになっているのが原因でした。

login.tailscale.com/admin/acls で確認できます。デフォルトはおおよそ次の形でした。

```json
{
  "acls": [
    { "action": "accept", "src": ["*"], "dst": ["*:*"] }
  ],
  "ssh": [
    {
      "action": "check",
      "src":    ["autogroup:member"],
      "dst":    ["autogroup:self"],
      "users":  ["autogroup:nonroot", "root"]
    }
  ]
}
```

つまり、自分のマシンに自分で SSH するときも 12時間ごとにブラウザ確認が走る、ということです。仕様ですし、セキュリティ的にもむしろ推奨設定だと思います。煩わしければ ACL を `"action": "accept"` に変えれば即接続になりますが、**サーバーが侵入されたときに横移動を止められなくなる** ので、1人運用でも `check` のまま使うのがおすすめです。

参考リンク:

- https://tailscale.com/kb/1018/acls
- https://tailscale.com/kb/1193/tailscale-ssh

## まとめ

今回やったことを振り返ると以下です。

- 公式インストーラ → `tailscale up --ssh` → ブラウザ認証、の 3 ステップで RPi5 へのリモート SSH が完成しました
- ポート開放ゼロ、SSH 鍵管理ゼロ、ホスト名 `hostname` だけでどこからでも届くようになりました
- 常時稼働サーバーでは Key Expiry の無効化を忘れずに
- Tailscale の `check` モードは煩わしく感じても、運用上は残す価値ありでした
